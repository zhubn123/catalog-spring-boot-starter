# Contributing

感谢关注这个项目。

这份文档主要说明一个简单、清晰的协作习惯，方便后续按比较规范的开源方式维护。

## 提 issue

请优先使用仓库内置的 Issue Forms：

- `Bug Report`：用于缺陷、异常、回归问题
- `Feature Request`：用于新能力或接口改进建议
- `Refactor Task`：用于结构优化、解耦、治理和维护性整改

建议遵循下面几个原则：

- 一个 issue 只处理一个明确目标
- 先描述问题和场景，再描述方案
- 验收标准尽量可执行、可验证
- 如果是重构任务，请明确本次不做什么，避免范围失控

## 开始编码前

建议先确认以下内容：

- 是否已有相同或高度相似的 issue
- 是否已经明确目标语义和验收标准
- 是否会影响对外接口、数据库结构或示例工程

## Pull Request 建议

建议每个 PR 保持单一主题，并在描述中说明：

- 背景
- 主要改动
- 风险点
- 验收方式

如果变更涉及接口、配置、数据库或发布流程，请同步更新相关文档。

## 本地验证建议

提交前至少完成以下检查：

```bash
mvn -DskipTests package
mvn test
```

如果是 sample、发布流程或数据库相关改动，建议补充对应的手工验证说明。

## 维护性任务建议

对于治理类任务，推荐使用 `docs/issue-backlog.md` 中的格式整理任务，再复制到 GitHub 创建 issue。
