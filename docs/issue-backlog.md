# Issue Backlog

这份文档用于沉淀“可直接复制到 GitHub”的任务单。

推荐使用方式：

1. 在 GitHub 点击 `New issue`
2. 选择 `Refactor Task`、`Bug Report` 或 `Feature Request`
3. 参考下面对应条目的标题和内容，粘贴到表单中
4. 一条 issue 只处理一个任务

---

## 1. [Refactor] 去掉路径批量更新对多语句 SQL 的依赖

### 背景

当前路径批量更新依赖 Mapper 中拼接多条 `update` SQL。这通常依赖 JDBC 驱动或连接参数允许多语句执行，存在明显的环境耦合和运行风险。

### 目标

消除对多语句 SQL 的隐式依赖，使节点路径更新在默认 MySQL 环境下也能稳定运行。

### 改动范围

- `catalog-spring-boot-autoconfigure` 中的 `CatalogNodeMapper.xml`
- `catalog-core` 中与路径更新相关的服务逻辑

### 验收标准

- 节点移动、重命名后的路径更新可以正常完成
- 不依赖 `allowMultiQueries` 等特殊 JDBC 参数
- 本地构建和测试通过

### 不在本任务范围内

- 不调整 REST API 设计
- 不重构整套目录树建模方式

## 2. [Refactor] 将绑定操作调整为幂等语义

### 背景

当前重复绑定目录与业务对象时，容易暴露底层数据库异常，不利于对外提供稳定接口语义。

### 目标

让 `bind`、`batchBind` 等操作具备幂等性，重复提交时返回稳定结果。

### 改动范围

- `catalog-core` 中绑定相关服务逻辑
- `catalog-spring-boot-autoconfigure` 中关系表 Mapper

### 验收标准

- 相同参数重复调用不会抛数据库层异常
- 调用结果可预测，文档语义清晰
- 相关测试通过

### 不在本任务范围内

- 不调整业务对象与目录节点的多绑定模型

---

## 3. [Feature] 明确并修正业务对象路径查询语义

### 背景与场景

当前业务对象支持多绑定，但路径查询接口只返回单条路径，容易造成语义混乱。

### 当前问题

调用方无法区分“只绑定一个目录”和“绑定多个目录但接口只返回第一条”的情况。

### 建议方案

新增多路径查询接口，或明确旧接口只表示“主路径”，并同步更新文档和命名。

### 备选方案

如果业务上不需要多绑定，则反向收紧模型和约束。

### 验收标准

- 接口语义、命名、文档一致
- 多绑定场景有明确表现
- 相关查询测试通过

---

## 4. [Refactor] 修正树查询排序规则

### 背景

当前树查询按 `path` 字符串排序，目录较多时会出现展示顺序与业务排序不一致的问题。

### 目标

统一树和子树查询的排序规则，使同级节点按 `sort` 字段稳定返回。

### 改动范围

- 树查询相关 Mapper SQL
- 树查询相关 Service 逻辑

### 验收标准

- 同级节点按 `sort` 升序返回
- `10` 不会排在 `2` 前面
- 示例接口和文档表现一致

### 不在本任务范围内

- 不调整前端展示逻辑

---

## 5. [Refactor] 将 CatalogController 的批量接口升级为 DTO

### 背景

当前批量操作大量依赖逗号分隔字符串，不利于参数校验、文档生成和后续演进。

### 目标

把批量新增、批量绑定、移动等接口改为结构化请求模型。

### 改动范围

- `catalog-spring-boot-autoconfigure` 中 `CatalogController`
- 新增请求 DTO
- README 中接口示例

### 验收标准

- 不再要求调用方手动拼接 CSV 字符串
- 关键参数具备明确校验
- 接口文档或示例同步更新

### 不在本任务范围内

- 不改变核心业务语义

---

## 6. [Refactor] 统一 tree 接口命名与返回模型

### 背景

当前 `/tree` 名称看起来像返回嵌套树，但实际更接近平铺列表，存在认知偏差。

### 目标

让接口命名和返回结构保持一致，避免误导调用方。

### 改动范围

- `CatalogController`
- `CatalogServiceImpl`
- README 接口说明

### 验收标准

- 如果保留 `/tree`，则返回真正的树形结构
- 如果继续返回平铺列表，则接口命名改为更准确的表达
- 文档与实现一致

### 不在本任务范围内

- 不做复杂前端适配

---

## 7. [Refactor] 补统一异常和参数校验

### 背景

当前接口存在直接抛运行时异常或数据库异常的情况，调用方处理成本高。

### 目标

建立最小统一错误返回和参数校验机制。

### 改动范围

- 控制器层
- 异常定义和异常处理
- 必要的参数校验注解或逻辑

### 验收标准

- 参数错误、节点不存在、重复绑定等场景能稳定区分
- 接口失败时不直接暴露底层数据库异常

### 不在本任务范围内

- 不做完整的国际化错误码体系

## 8. [Refactor] 全面切换为构造注入

### 背景

当前多个核心类依赖字段注入，降低了测试便利性和装配清晰度。

### 目标

统一采用构造注入，提高可测试性和代码可维护性。

### 改动范围

- 自动配置类
- 控制器
- 核心服务实现

### 验收标准

- 去除核心类中的字段注入
- 不依赖 Spring 容器也能构造核心服务进行单测

### 不在本任务范围内

- 不强制大规模拆分类职责

## 9. [Refactor] 拆分 CatalogServiceImpl 职责

### 背景

当前 `CatalogServiceImpl` 集中承担节点管理、关系绑定、查询和排序修正等多类职责，类复杂度偏高。

### 目标

按职责拆分服务实现，降低修改风险，便于后续扩展和测试。

### 改动范围

- `catalog-core` 中服务层实现
- 相关接口和装配关系

### 验收标准

- 单个服务类职责更聚焦
- 查询与命令逻辑有清晰边界
- 原有对外接口行为保持兼容

### 不在本任务范围内

- 不一次性重构全部包结构

## 13. [Refactor] 为核心目录能力补回归测试

### 背景

当前核心逻辑涉及树结构、排序、绑定和路径更新，但缺少足够的自动化回归保护。

### 目标

为高风险逻辑建立基础测试网，降低后续重构风险。

### 改动范围

- `catalog-core` 测试目录
- 必要时补充 `catalog-spring-boot-autoconfigure` 集成测试

### 验收标准

- 覆盖新增节点、移动节点、删除节点、重复绑定、多路径查询、树顺序等核心场景
- `mvn test` 能稳定执行

### 不在本任务范围内

- 不追求一次性覆盖全部边界用例

## 14. [Refactor] 整理发布流程为独立 release 约定

### 背景

当前发布相关逻辑与日常构建耦合较紧，不利于开发和维护。

### 目标

将发布动作、签名和文档说明拆分成更清晰的 release 约定。

### 改动范围

- 根 `pom.xml`
- GitHub Actions workflow
- README 或发布说明文档

### 验收标准

- 日常本地构建不必关注发布细节
- 正式发布步骤清晰、可重复执行
- 发布前检查项可落地执行

### 不在本任务范围内

- 不更换仓库发布平台
